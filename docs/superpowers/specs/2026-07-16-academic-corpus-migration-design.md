# Academic Corpus Migration Design

**Date:** 2026-07-16
**Status:** Approved for implementation
**Product:** Parallel Worlds

## 1. Purpose

Parallel Worlds currently contains 62 historical lines. Sixteen lines have record-level evidence and `reviewStatus: reviewed`; forty-six inherited lines remain `legacy`. The next academic release will establish a reproducible audit system, strengthen the chronology schema, and migrate the first high-visibility group of legacy polities without pretending that the remaining corpus has already been checked.

This release is the first independently testable milestone of the wider 62-line migration. Later releases will cover the Americas and Africa, the rest of Asia and Oceania, and finally religious and philosophical traditions.

## 2. Release scope

### 2.1 Included

- Generate a deterministic academic audit for every track, period, event, and source.
- Distinguish blocking evidence failures from migration warnings.
- Extend dating metadata with confidence, calibration, alternative models, and dispute notes.
- Reject generic homepages as evidence for reviewed records.
- Make audit status visible in details and exportable as JSON.
- Migrate the following first-batch legacy lines to record-level evidence:
  - Egypt;
  - Akkad;
  - Babylonia;
  - Assyria;
  - Hittites;
  - Persia;
  - Greece;
  - Rome;
  - Byzantium.
- Preserve complete Russian, English, and Simplified Chinese copy.
- Add exact institutional or scholarly sources for every migrated period and event.

### 2.2 Deferred

- Migration of the remaining African, American, Asian, Oceanian, and religious lines.
- Moving the canonical boundary earlier than 20,000 BCE.
- Runtime access to external APIs or automatic historical claims generated by AI.
- Full radiocarbon recalibration in the browser.

## 3. Evidence policy

### 3.1 Source roles

Sources keep the existing A/B/C tiers and add a `role` vocabulary at the record link level:

- `absolute-chronology`: radiocarbon, dendrochronology, astronomical anchors, or dated inscriptions;
- `periodization`: a scholarly definition of the period and its conventional bounds;
- `event-attestation`: a source directly supporting an event;
- `interpretive-synthesis`: a scholarly synthesis used for explanatory copy.

A reviewed record needs at least one exact source supporting its interval or event. Interpretive synthesis alone cannot turn an uncertain date into an exact date.

### 3.2 Preferred source order

1. Peer-reviewed chronometric research or excavation reports.
2. Scholarly museum publications and exact institutional chronologies.
3. Official UNESCO nomination dossiers with explicit chronology.
4. University-press and specialist academic syntheses.
5. Reputable educational overviews for orientation only.

Search results, general homepages, Wikipedia, blogs, and unsourced lists are never evidence sources.

### 3.3 Methodological anchors

- Period definitions may be cross-checked against PeriodO: https://perio.do/
- Radiocarbon normalization uses the relevant IntCal20 family publication and preserves the source notation: https://www.intcal.org/publications.html
- Dynastic Egypt uses the Bayesian radiocarbon chronology by Bronk Ramsey et al.: https://doi.org/10.1126/science.1189395
- Early Egypt uses Dee et al.: https://doi.org/10.1098/rspa.2013.0395
- Earlier second-millennium Mesopotamia uses Manning et al.: https://doi.org/10.1371/journal.pone.0157144

These anchors define method, not a universal chronology for unrelated regions.

## 4. Dating model

Existing fields remain compatible:

```js
{
  precision: 'range',
  basis: 'historical',
  original: 'ca. 2350–2150 BCE'
}
```

Reviewed records may add:

```js
{
  confidence: 'high',
  calibrationCurve: 'IntCal20',
  model: 'preferred',
  alternatives: [{
    id: 'middle-chronology',
    start: -1792,
    end: -1750,
    label: 'Middle chronology'
  }],
  disputeNote: 'Alternative chronologies shift the accession sequence.'
}
```

Rules:

- `confidence` is `high`, `medium`, or `low`.
- `calibrationCurve` is required only when a source record is explicitly normalized from radiocarbon evidence.
- `alternatives` must contain valid non-zero historical intervals and stable IDs.
- `disputeNote` is required when alternatives exist or `precision` is `disputed`.
- A conventional period range is not silently relabeled as an exact range.
- Year zero remains invalid everywhere.

## 5. Audit architecture

Create `academic-audit.js` as a pure module. It consumes the localized-independent canonical dataset and returns:

```js
{
  generatedFrom: 'parallel-worlds-data-v1',
  summary: {
    tracks: 62,
    reviewedTracks: 25,
    legacyTracks: 37,
    blockingIssues: 0,
    warnings: 42
  },
  coverage: {
    periods: { total: 217, sourced: 69, dated: 69 },
    events: { total: 161, sourced: 50, dated: 50 }
  },
  tracks: [],
  sources: [],
  issues: []
}
```

Issue severities:

- `error`: invalid reviewed record, invalid year/range, unknown source, missing localization;
- `warning`: legacy record without record-level evidence, generic inherited source, broad legacy type;
- `info`: optional author, locator, or alternative model not yet supplied.

The module must be deterministic: it contains no current timestamp, network access, or environment-specific path.

To keep the report useful, legacy coverage produces one `legacy-track` warning per line rather than one warning for every inherited period and event. Each generic inherited source produces one additional warning. Record-level gaps remain visible in the per-track coverage counts.

`scripts/build-academic-audit.mjs` serializes the report to `academic-audit.json`. Tests compare stable fields rather than relying on a generated timestamp.

## 6. Data migration

### 6.1 Migration mechanics

Keep the current split:

- `data.js` retains the inherited corpus and localization factory.
- `academic-data.js` owns reviewed source records and patch definitions.
- A patch replaces the type, review state, copy, periods, events, and source list for a migrated line.

This avoids a high-risk rewrite of the full dataset while making every migrated record structurally identical to the existing reviewed corpus.

### 6.2 First-batch source packs

Each migrated line receives exact sources rather than the old `met`, `british`, or other homepage IDs.

- Egypt: Science radiocarbon chronology, early Egypt Bayesian model, exact Met/British Museum period chronologies.
- Akkad: Met essay `The Akkadian Period (ca. 2350–2150 B.C.)`.
- Babylonia: exact Met essays or collection chronologies for Old Babylonian, Kassite, and Neo-Babylonian phases, supported by Mesopotamian chronology research where relevant.
- Assyria: exact Met essays and institutional chronologies for Old, Middle, and Neo-Assyrian phases.
- Hittites: exact Met or institutional Hittite chronology plus the Mesopotamian/Anatolian chronometric anchor where applicable.
- Persia: exact Encyclopaedia Iranica or Met chronologies for Achaemenid, Parthian, and Sasanian periods.
- Greece: exact Met Greek chronological essays for Bronze Age, Archaic, Classical, and Hellenistic phases.
- Rome: exact Met chronological essays for Republican, Early Imperial, Late Imperial, and western post-imperial boundaries.
- Byzantium: Met scholarly chronology for Early, Middle, Latin occupation, and Late Byzantine phases.

One exact source may support multiple records only when that page explicitly supplies the relevant intervals.

## 7. Interface

The ordinary experience stays compact. Evidence improvements appear in details:

- review badge remains visible;
- details show confidence, calibration curve, chronology model, dispute note, and alternatives when present;
- the method section links to `academic-audit.json`;
- a new filter option `reviewed` shows only fully reviewed lines;
- CSV adds confidence, model, alternatives, and dispute note after existing evidence columns.

The audit JSON is a static asset and does not require a runtime request for the core site to function.

## 8. Validation

Automated tests must cover:

- deterministic audit output;
- severity classification for reviewed and legacy records;
- exact-source coverage metrics;
- generic legacy source warnings;
- dating confidence and alternative-model validation;
- rejection of year zero inside alternatives;
- complete RU/EN/ZH copy for migrated records;
- every first-batch period and event carrying `sourceIds` and dating metadata;
- source URLs resolving to exact HTTPS pages or DOIs;
- reviewed-only filtering;
- extended CSV fields;
- inclusion of `academic-audit.js` and `academic-audit.json` in Pages packaging.

Browser checks cover all three locales, reviewed-only filtering, one disputed chronology, one alternative-model detail, and mobile evidence layout.

## 9. Release gates

This milestone is complete when:

- the academic audit is reproducible locally and in CI;
- all reviewed records pass strict validation;
- the nine first-batch legacy lines are reviewed;
- legacy count decreases from 46 to 37;
- no migrated record depends on a generic homepage;
- the UI and CSV expose added evidence without breaking older records;
- tests, static validation, browser checks, and GitHub Pages deployment pass.

The release must state clearly that 37 lines remain legacy and will be migrated in later source-backed batches.
